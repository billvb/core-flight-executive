CFS Record File Conventions

Formatting Conventions

  • The last character in every file should be newline.
  • Do not use tabs.
  • Use 2 spaces for each level of indentation.
  • Prototype names begin with uppercase letters.
  • Instance names begin with lowercase letters.
  • Anything that comes to us from RDL in all uppercase should be converted to all lowercase.
  • For longer names, consider making them camelcase for readability, but keeping the first letter lowercase.
  • Command packet names should be lowercase or contain underscores, but should not be camelcase.
  • Command Enumeration names should be all lowercase.
  • Dsc record instance names may begin with uppercase characters since they double as the displayString.
  • Put a comma followed by a space between elements in a list on the same line.
  • If a line is too long, put one item on each line and don’t use commas.
  • Always keep a single property on the line with the record name.
  • For 2 or 3 properties, use best judgement.  Generally keep length and padding together on 1 line.
  • For more than 3 properties, use multiple lines.
  • Don’t use commas if you’re creating a lined-up block as in the properties of the PortConnection record in a Graph’s connections property.
  • Generally, do not put spaces around =
  • Except when = is followed by {, then put spaces before and after =.
  • Put spaces before and after { and }
  • Render {} without a space inside, but with a space before and after.
  • Do not put a space after } if it ends the line.
  • Do not put { on a line by itself, but do put } on a line by itself when {} encloses multiple lines.
  • Use a consistent order of properties and overrides in all cases.
  • Keep offset and offsetFrom together.
  • List prePadding, lengthInBits, and postPadding in that order.
  • Keep the foregoing together (don’t put generateMnemonic between them, for example)
  • Never use /**--> */ but use /** */ instead.

File and Naming Conventions

  • Create a prototype of CfsCommand inside an application’s prototype System where there are many commands with a common applicationId value.
  • Name it XXCommand where XX is the abbreviation for the application.
  • Avoid redundancy in names.
  • System aa { U aa_bbbb {} U aa_cccc {} }  Instead, use: System aa { U bbbb {} U cccc {} }
  • Do not include things like “tbl” in table field names.
  • Look for and eliminate redundancy between command names and command field names where possible.
  • Consider the inherent comprehensibility of fully-qualified field names.
  • Do not repeat the property lengthInCharacters=n for multiple items holding like things.
  • cfs.rec defines FileName, AppName, and others for this.
  • Define app-specific prototypes where appropriate or add new prototypes to cfs.rec.
  • Files and tables:
  • Use the cfs.File prototype to define files and cfs.Table prototype to define tables.
  • In an application System, create System table {} to hold table definitions and System file {} to hold file definitions.
  • Define a TelemetrySource for each file or table.
  • If it makes sense, name the source the same as the file or table, and enclose all such sources in a System sources {} inside the System table {} or System file {}.
  • Separate commands, telemetry, files and tables with comment blocks such as:

  /* ------------------- */

  /* COMMAND DEFINITIONS */

  /* ------------------- */

  • Files created by the Hammers Company carry this statement in each file:

*/

Notes on the current state of the CFS Record files

  • These definitions are largely untested.
  • Many packet and field names are taken from the RDL files included with the CFS.
  • While a consistent naming convention was not always followed in the RDL, we have attempted to improve consistency based on the conventions above.  Nevertheless:
  • There is a mix of camelcase and lowercase with underscores.
  • Certain common fields may not always carry the same names.
  • RDL names in all uppercase were converted to all lowercase; camelcase names all now begin with lowercase letters.
  • Most command definitions reference a Verification record which provides the accepted-by-application level of command verification.
  • We will add the executed level of command verification where possible.
  • The file CfsSpacecraft.rec-example provides a functional implementation of CFS for a single processor on a single spacecraft.
